Introducción:
El presente informe documenta el diseño y la implementación del manejador de recursos distribuidos para High-Performance Computing. El cluster HPC es simulado. El objetivo principal del proyecto es construir un nodo autónomo capaz de interoperar con otros nodos en la red, abordando los desafíos que conlleva la sincronización distribuida, el riesgo de deadlocks, y demás. 

Para resolver esta complejidad, la arquitectura del nodo se divide en dos componentes especializados que se comunican mediante una interfaz local:
Resource Manager Agent (C): Actúa como el administrador de los recursos físicos del nodo. Utiliza epoll para gestionar de forma concurrente y no bloqueante múltiples conexiones TCP. Además de manejar los recursos locales, es el encargado de interactuar con agentes remotos y de descubrir dinámicamente otros nodos en la red mediante broadcast UDP.
Job Scheduler (Erlang): Funciona como el cerebro planificador. Se conecta al agente C para generar la carga de trabajo simulada, tomar decisiones sobre qué recursos solicitar en el clúster y, fundamentalmente, aplicar la estrategia lógica para prevenir o detectar los deadlocks distribuidos.

A lo largo del informe se detalla cómo se ensamblan ambos módulos utilizando los protocolos definidos, las estructuras de datos empleadas para el manejo de estado y la estrategia adoptada para garantizar que el clúster opere de manera fluida y tolerante a bloqueos.

Planificador en Erlang:
	El programa comienza en el módulo main, en la función main. Primero se crea y registra el logger globalmente, en el cual registramos denegaciones de recursos a trabajos, timeouts, errores del scheduler y demás. 

El módulo logger tiene la función de creación del archivo .log, la escritura y cierre del mismo.

	Una vez creado el logger se intenta la conexión con el servidor del agente de C, en caso de que no se pueda lograr la conexión el programa lo intentará nuevamente con una cierta espera. Por otro lado si esta se logra establecer, se genera el proceso scheduler, quien espera la creación del oyente del socket de C. Una vez creado este último, el scheduler comienza a trabajar.
	El proceso principal, una vez que crea el scheduler, se queda esperando a que el proceso scheduler le envíe una solicitud para cerrar el sistema, y si esta llega la conexión se intenta hacer nuevamente.

	Antes de hablar del scheduler, que es el proceso principal, hablemos del oyente del socket en C. El oyente del socket del Agente se encuentra en el módulo tcpClient, este módulo es el que tiene todas las funciones encargadas de la conexión y envío/ recibo de mensajes con el agente de  C, además tiene un parseo inicial de los mensajes de C.

	El proceso scheduler inicia solicitando los nodos que hay en la red, luego crea dos procesos, un generador de trabajos y un actualizador de nodos. El primero enviará solicitudes de creación de trabajo y el segundo solicitará la actualización de los nodos, ambos cada cierto tiempo. Con esto se ejecuta la función del scheduler propiamente dicho. Este permanece constantemente esperando mensajes tanto del agente de C, como las interacciones con el trabajo simulado y la petición de nodos.
	
	En la sección de interacción con nodos, si se encuentra un error al realizar el parseo de los nodos, se utiliza la lista vieja obtenida hasta ese momento. En caso de éxito en el parseo, se calculan los recursos totales del clúster con la función del módulo manejoRecursos correspondiente. El objetivo del cálculo de recursos totales es para que el trabajo simulado no se exceda en la toma de recursos.

	Hablemos ahora de las interacciones con el trabajo simulado, si el scheduler obtiene una solicitud para generar trabajo se crea un proceso que actuará como Job, este proceso usa funciones del módulo jobWorker. Este módulo genera cantidades aleatorias de recursos a pedir y espera la respuesta del scheduler con el mensaje del agente, es decir si el Job solicitado dio granted, denied o timeout. Además este módulo anota en el log lo que pasa con ese Job. Por otro lado el job simulado pide al scheduler recursos, quien mediante la función asignarNodos del módulo manejoRecursos lo gestiona, dando una lista de requerimientos concretos para enviar al agente de C. Por último si el Job termina correctamente envía jobTerminado y el scheduler lo recibe. Ahora bien si hay un cierre inesperado del proceso del Job simulado, se lo atrapa para poder limpiar el mapa.

	Por último, se tiene la interacción del scheduler con el agente de C. Este espera los mensajes del agente y todo lo que hace es “redirigir” el mensaje al trabajo simulado en caso de que la respuesta sea correcta o si la respuesta es distinta, ya sea por algún comando desconocido o un error en la conexión con el oyente, maneja la situacion.

Decisiones de diseño del planificador:
A lo largo del planificador tomamos una serie de decisiones que vale la pena justificar, porque explican por qué el código está estructurado como está y no de otra forma.

La primera y más importante es que cada job es un proceso independiente, creado con spawn_monitor. La razón es que el scheduler necesita atender mensajes de múltiples fuentes en paralelo: respuestas del agente C, nuevos pedidos de trabajo y actualizaciones de nodos. Si un job esperara su granted, denied o timeout dentro del propio scheduler, bloquearía a todos los demás. Al tener un proceso por job, el receive bloqueante vive en el worker y el scheduler nunca se frena. La alternativa que descartamos era manejar todos los jobs dentro del scheduler con un mapa de estados y timeouts manuales, que resulta más complejo, más frágil y contrario al modelo de actores.

Ligado a esto, elegimos spawn_monitor para los workers pero spawn_link para los servicios internos (el generador de trabajos y el actualizador de nodos). La regla que seguimos es simple: usamos link cuando el proceso hijo no tiene sentido sin el padre —si el scheduler muere, no tiene sentido seguir generando jobs, así que el link los arrastra automáticamente— y usamos monitor cuando el padre debe sobrevivir al hijo. Si un worker crashea, el scheduler recibe un {'DOWN', ..., Razon}, limpia el mapa y sigue vivo; no debe morir porque un job falló.

Otra decisión de fondo es la separación de responsabilidades entre worker y scheduler. Los workers piden cantidades abstractas de la forma {CantCpu, CantMem, CantGpu} y es el scheduler quien traduce eso a asignaciones concretas del estilo {IP, "cpu", N} antes de mandárselas al agente C. El motivo es que los workers no conocen la topología del cluster —qué nodos existen ni cuánto tienen—, eso es responsabilidad del scheduler, que mantiene la lista de nodos actualizada. Una consecuencia directa y deseable es que, si cambia la lista de nodos, los jobs en vuelo no se ven afectados, porque cada uno ya mandó su solicitud con nodos concretos.

Para traducir la demanda a nodos concretos usamos un algoritmo voraz (asignarVoraz). Cuando un job pide, por ejemplo, 7 CPUs y ningún nodo tiene 7 pero dos nodos tienen 4 y 5, el algoritmo reparte la demanda entre ambos: @nodoA:cpu:4 @nodoB:cpu:3. Una búsqueda simple del tipo "busco un nodo con capacidad suficiente" descartaría este job aunque el cluster tenga recursos totales de sobra. El voraz recorre los nodos en orden aleatorio tomando de cada uno lo que puede hasta cubrir la demanda; si no logra cubrir la demanda completa de algún recurso devuelve insuficiente, y en ese caso el scheduler deniega el job localmente sin siquiera molestar al agente C.

Internamente representamos los recursos como una tupla posicional {Cpu, Mem, Gpu} y no como mapas ni listas de propiedades. El orden es fijo y conocido por todos los módulos, así que el acceso es por pattern matching directo, sin overhead de lookup. Los nombres "cpu", "mem" y "gpu" solo aparecen en los bordes del sistema, al parsear la respuesta de GET_NODES y al construir el JOB_REQUEST. El trade-off consciente es que agregar un cuarto tipo de recurso obligaría a tocar la tupla en todos los módulos, algo que priorizamos frente a la extensibilidad porque el enunciado define exactamente tres recursos.

En cuanto a la robustez de la conexión, el main no termina ante una desconexión: reintenta con backoff exponencial, esperando el doble de tiempo entre intentos con un techo de 30 segundos. El cierre limpio previo a reconectar sigue una secuencia ordenada: el scheduler recibe un errorRecvAgente del oyente TCP, mata todos los workers con exit(Pid, shutdown) para que no queden bloqueados en un receive esperando mensajes que ya no van a llegar, notifica al main con cerrarSistema, y se mata a sí mismo con una razón no-normal, lo que por el spawn_link arrastra al generador de trabajos y al actualizador de nodos. Elegimos backoff exponencial y no un intervalo fijo porque un intervalo fijo bajo genera reconexiones agresivas si el agente C tarda en levantarse, y el techo de 30 segundos evita que la espera crezca sin límite.

El logging lo resolvimos con un único proceso registrado globalmente como schedulerLogger, que abre el archivo y atiende mensajes {log, Linea} en serie. Si múltiples workers escribieran directamente al mismo archivo desde procesos distintos, el texto quedaría intercalado; el proceso logger serializa los accesos sin necesidad de locks explícitos, que es el modelo de actores aplicado a la I/O. Lo registramos con un nombre y no pasamos su PID por parámetro para no ensuciar todas las firmas de funciones: cualquier proceso puede loguear con schedulerLogger ! {log, ...} sin coordinación extra.

La actualización de nodos la hace un proceso separado que envía GET_NODES cada 15 segundos, con independencia de la actividad del scheduler. Elegimos 15 segundos porque el enunciado considera caído a un nodo que no anuncia presencia durante ese lapso, así que refrescar con ese mismo intervalo garantiza que el scheduler nunca trabaje con una lista más vieja que un ciclo de anuncio UDP. Preferimos un proceso dedicado antes que un send_after que el scheduler tendría que reprogramar en cada ciclo: el proceso es autónomo y, si el scheduler está ocupado procesando mensajes, el timer no se atrasa.

Finalmente, el scheduler ignora la generación de trabajo cuando ya hay MAX_JOBS jobs en curso. Sin ese tope, el generador acumularía decenas de workers simultáneos pidiendo recursos, y con clusters chicos de dos o tres nodos todos quedarían denegados o en espera, saturando el sistema de mensajes sin progresar. El valor vive en config.hrl, que es el único punto de cambio para ajustar la carga.

Un último detalle de la capa de red del planificador: la conexión con el agente C usa {packet, line} en lugar de modo raw. Con modo raw, si el agente C manda dos respuestas seguidas muy rápido (por ejemplo JOB_GRANTED 1\nJOB_GRANTED 2\n), Erlang podría recibirlas en un solo recv como un bloque concatenado. Con {packet, line} el kernel garantiza que cada recv devuelve exactamente una línea completa, sin importar cómo llegaron los bytes por TCP, y el parser puede asumir siempre un comando por llamada.

Estrategia elegida contra Deadlock:
Se decidió tomar los recursos como fue convenido en el curso: CPU, Memoria, GPU. En un caso perfecto de que todos los nodos tomen los recursos de la misma manera, esta podría ser la única estrategia contra deadlock implementada, sin embargo, al no estar seguros de que todos los nodos de los distintos grupos los tomen de la misma manera se implementó un timeout, evitando el bloqueo del sistema a cambio de la posible ocurrencia de livelock.

El ejemplo de deadlock se evita ya que desde el vamos dos Jobs nunca tomarían recursos de la forma CPU -> GPU y otro GPU -> CPU, ambos harían CPU -> GPU. Esto elimina la espera circular. Como hemos mencionado anteriormente este es un caso perfecto, supongamos entonces que el Job1 pide CPU y despues GPU, tanto que el Job2 pide GPU y después CPU, en este caso si el Agente A concede CPU a Job1 y el Agente B concede GPU al Job2 entonces si nuestro agente es el A, cuando note que se esta tardando mas del tiempo estipulado en dar la GPU, enviara un timeout al Job1, quien desde erlang, soltará los recursos y intentara volver a ejecutarse después de un cierto tiempo.

En concreto, la estrategia es detección por timeout combinada con liberación de recursos y reintento con espera aleatoria. Descartamos la prevención estática por ordenamiento de recursos porque exigiría un orden canónico de IPs y recursos acordado entre todos los equipos, incluidos los de otros grupos; también descartamos la detección por grafo de espera, que requiere un estado global distribuido difícil de mantener sin un coordinador central, y los esquemas tipo Wound-Wait o Wait-Die, que necesitan timestamps globales de inicio de cada job difíciles de sincronizar entre nodos independientes. El timeout con backoff aleatorio, en cambio, es simple, no agrega estado adicional y se resuelve localmente en cada agente y cada worker; la aleatoriedad rompe la simetría sin coordinación global, y el modelo de actores de Erlang la favorece porque cada job es un proceso que puede dormirse y reintentar sin bloquear a nadie.

La detección vive en el agente C, que usa un timerfd que dispara cada 15 segundos (en el handler del timer de mantenimiento) y aplica el timeout en dos planos distintos. Del lado dueño del recurso, si un RESERVE que otro agente encoló en este nodo lleva más de 30 segundos (TIEMPO_ESPERA_RESERVA) sin resolverse, se lo desencola en silencio: es higiene de cola, libera el recurso para otros pedidos y no notifica a nadie. Del lado coordinador, si una PeticionMulti —un job que este agente coordina y para el que mandó RESERVE a otros nodos— lleva más de 30 segundos sin completarse (respondidos < total), el agente hace rollback enviando RELEASE a todos sus nodos y avisa a su planificador Erlang con JOB_TIMEOUT.

Un punto clave del diseño es que no agregamos ningún mensaje de timeout entre agentes: el coordinador detecta el vencimiento por sí mismo mirando el timestamp de su PeticionMulti, y el protocolo entre agentes C sigue siendo únicamente RESERVE, GRANTED, DENIED y RELEASE. Esto nos permite reservar el DENIED para el rechazo genuino de un nodo, es decir un pedido imposible (recurso desconocido o cantidad mayor a la capacidad total), y no confundirlo con una situación de congestión.

Esta distinción se traslada al contrato con Erlang y define cómo reacciona el worker. Un JOB_DENIED es un fallo permanente: el worker lo loguea como [DENIED] y finaliza el job sin reintentar, porque un pedido imposible no mejora por reintentarse. Un JOB_TIMEOUT es congestión o deadlock: el worker lo loguea como [TIMEOUT], espera una cantidad aleatoria de tiempo (rand:uniform(?MAX_RANDOM_ESPERA)) y reintenta el job desde cero, generando nuevas cantidades y mandando un nuevo JOB_REQUEST. Por eso el timeout del coordinador se manda como JOB_TIMEOUT y no como JOB_DENIED: es lo que habilita el reintento que rompe el deadlock. La espera aleatoria es el mecanismo central, porque si dos jobs en deadlock esperaran el mismo tiempo fijo volverían a colisionar; la aleatoriedad hace estadísticamente improbable que ambos reintentos vuelvan a chocar.

Para ver la estrategia completa conviene seguir un ejemplo con dos nodos. Supongamos el Nodo A (192.168.1.10) con 2 de CPU, 8 GB de memoria y 0 de GPU, y el Nodo B (192.168.1.11) con 2 de CPU, 4 GB de memoria y 1 de GPU. El Job 1001, del Erlang de A, solicita cpu:2 del Nodo A y gpu:1 del Nodo B; el Job 2001, del Erlang de B, solicita gpu:1 del Nodo B y cpu:2 del Nodo A. En una primera fase las solicitudes se cruzan y ambas se conceden parcialmente:

Agente A -> Nodo A:  RESERVE 1001 cpu 2  ->  GRANTED   (la cpu de A queda en 0)
Agente B -> Nodo B:  RESERVE 2001 gpu 1  ->  GRANTED   (la gpu de B queda en 0)

En la segunda fase aparece el deadlock, porque cada job pide el recurso que el otro ya tiene tomado:

Agente A -> Nodo B:  RESERVE 1001 gpu 1  ->  ENCOLADO  (gpu ocupada por el Job 2001)
Agente B -> Nodo A:  RESERVE 2001 cpu 2  ->  ENCOLADO  (cpu ocupada por el Job 1001)

El Job 1001 tiene cpu en A y espera gpu en B, mientras el Job 2001 tiene gpu en B y espera cpu en A: espera circular. En la tercera fase, pasados 30 segundos, ninguna de las dos PeticionMulti se completó, así que expiran del lado coordinador:

Agente A: expira la PeticionMulti 1001 -> RELEASE a todos sus nodos -> JOB_TIMEOUT 1001 al Erlang A
Agente B: expira la PeticionMulti 2001 -> RELEASE a todos sus nodos -> JOB_TIMEOUT 2001 al Erlang B

En paralelo, cada nodo desencola en silencio el RESERVE ajeno que tenía vencido, y los recursos quedan libres: la cpu de A vuelve a 2 y la gpu de B vuelve a 1. En la cuarta fase los workers reciben el timeout, esperan un tiempo aleatorio distinto y reintentan; con alta probabilidad uno reintenta antes, adquiere todos sus recursos y libera el camino para el otro, con lo que el deadlock se rompe. En el log queda registrada la secuencia de [TIMEOUT] de ambos jobs seguida de los [GRANTED] de los reintentos exitosos.

De esta estrategia se desprenden sus propiedades. En cuanto a liveness, cuando dos jobs están en deadlock el timeout del coordinador los expira, libera sus recursos y avisa JOB_TIMEOUT, y los workers reintentan con backoff aleatorio rompiendo la simetría con alta probabilidad; el riesgo teórico es el livelock, es decir reintentos que vuelven a colisionar, mitigado por la aleatoriedad de la espera. En cuanto a safety, no hay doble asignación: el coordinador manda RELEASE a todos sus nodos antes de avisar el timeout y los nodos dueños desencolan sus pedidos vencidos, y un GRANTED tardío que llegue después de expirar la petición se ignora, porque la petición ya no está en la tabla. Y todo esto ocurre sin coordinación global: cada agente y cada worker actúan de forma independiente, sin mensajes de timeout entre agentes ni estado compartido fuera del agente C local.

Gestor de recursos en C:
La implementación principal del gestor de recursos en C es gestor_estado.c, esta representa la capa de abstracción mas alta, la misma se encuentra implementada con los mecanismos de sincronización necesarios para permitir que el epoll sea multihilos evitando race conditions. Su integración al mecanismo de comunicación se encuentra en controlador.c, este apartado parsea los mensajes recibidos para poder realizar correctamente las respectivas llamadas a función que gestor_estado le brinda.
El gestor de estado cuenta con las implementaciones estado.c (creación y destrucción del estado global del agente), nodos.c, recursos.c, peticiones.c y jobs.c. La primera se encarga de llevar registro de los nodos conectados (mediante una tabla hash y una lista simplemente enlazada) para esto implementa la funcionalidad de procesar announce añadiendo a la tabla un nodo nuevo o reiniciando el timestamp de uno ya existente, además implementa el get_nodes, volcando el correspondiente mensaje de salida en un buffer, por último también implementa el protocolo de desconección de aquellos nodos que no hayan realizado el announce en el tiempo correspondiente, la función correspondiente debe llamarse periódicamente desde el bucle del epoll para garantizar la actualización de la tabla.
recursos.c implementa la estructura de un recurso local, pudiendo ser cpu, gpu, mem y las solicitudes pendientes de un recurso. Implementa solo la creación y destrucción de su estructura, quedando la lógica de encolación de solicitudes para jobs.c. Este último, lleva registro de los jobs mediante una tabla hash y una lista, registra las asignaciones de recursos locales a los jobs, encolando la solicitud del recurso correspondiente si el mismo no está disponible en el momento, y en contrapartida liberando las solicitudes que ya lleven más de 30 segundos sin ser satisfechas, para esto también decidimos chequear las solicitudes encoladas periódicamente y de esa forma evitar un comportamiento lazy sobre las mismas, donde el programa no las denegará, por más que ya haya transcurrido el tiempo, hasta que llegue un receive. También registra las liberaciones de recursos, eliminando el registro del job si es que ya liberó todos los recursos que consumía. Implementa la liberación de los recursos consumidos por jobs de cierto nodo, para de esta forma poder manejar las desconecciones abruptas. Finalmente peticiones.c implementa las peticiones de múltiples recursos (PeticionMulti), de esta forma ante una petición rechazada (denied) permite liberar los recursos que ya fueron otorgados localmente o enviar el release de dichos recursos a los nodos correspondientes. Cada PeticionMulti guarda además su instante de creación: si el coordinador no la completa en el tiempo estipulado, un timer la expira, hace rollback (RELEASE a todos sus nodos) y envía JOB_TIMEOUT a Erlang para que reintente. Esto es necesario para evitar que los recursos queden reservados en un job que no se puede satisfacer y para romper deadlocks.
Tanto en el registro de los nodos activos como en el de los trabajos activos se almacena en una tablahash y en una lista para mantener la búsqueda de manera eficiente pero permitir su recorrido.

El agente está pensado en capas, donde cada una sabe lo mínimo de la de abajo: main se encarga del arranque, servidor.c es el transporte y el reactor de epoll, controlador.c es la capa de protocolo, gestor_estado.c es la fachada sobre el estado y modelo/ es el dominio (recursos, jobs, nodos y peticiones). El servidor no conoce el protocolo ni el estado, y el controlador no conoce el epoll salvo para pedirle "enviá esto" o "abrí una conexión"; el estado global lo define y lo usa el controlador.

Sobre esa base, gestor_estado.c funciona como una fachada thread-safe: casi todas sus funciones toman el lock del estado internamente, con la excepción de gestor_buscar_peticion y gestor_eliminar_peticion, que están pensadas para llamarse con el lock ya tomado y se documentan como tal. La función de reserva devuelve un código según lo que ocurra —1 si se concede (GRANTED), 0 si el pedido queda encolado y -1 si se rechaza (DENIED)—, y las funciones de liberación devuelven el recurso y reparten la cola FIFO de pendientes de forma voraz, concediendo a los siguientes en espera mientras alcancen los recursos. La expiración también está separada en los dos planos ya descritos: del lado dueño se desencolan en silencio los RESERVE locales vencidos, y del lado coordinador se expiran las PeticionMulti sin completar, invocando por cada una el callback que manda el RELEASE a sus nodos y el JOB_TIMEOUT a Erlang antes de destruirla. Completan la fachada las funciones de acceso al registro de nodos y sus conexiones cacheadas, y la de desconexión de un socket, que al caerse una conexión libera todos los recursos de sus jobs y reparte la cola.

Comunicación en C

La comunicación principal de nuestro agente con otros agentes C remotos se inicializa en main.c y el bucle de eventos corre en servidor.c. main.c se encarga de crear e inicializar la estructura EPOLL, los sockets de escucha TCP tanto local como de red, el socket de escucha UDP, los timers necesarios y el estado global del agente con sus recursos disponibles. Antes de lanzar los hilos, hace un descubrimiento inicial: emite un ANNOUNCE y espera unos segundos para que los agentes ya activos se anuncien y poblar así la tabla de nodos antes de empezar a atender. Luego, se agregan estos FDs al EPOLL para poder monitorear sus eventos, pero no se agregan como FD simplemente, sino que se agregan como una estructura la cual está especificada en cliente.h. 
Una vez todo iniciado, se crean 4 hilos que se encargaran de correr el bucle principal, donde atenderan los eventos del EPOLL y realizaran la lógica correspondiente dependiendo el evento. Es decir, si llego un cliente que se quiere conectar a los sockets de escucha este se acepta y se agrega su socket de cliente al EPOLL para luego ser atendido, si es el timer que marca el tiempo del announce se arma y envía el mensaje correspondiente, si es un cliente que envía mensajes estos se leen y luego se procesan, etc. 
En conjunto con estas, se utilizan las implementaciones sockets.c, cliente.c y controlador.c. 
La primera provee la abstracción e implementación de todo lo relacionado a la capa de red. Esta nos ofrece funciones para crear sockets de escucha y timers, conectar el agente a otro nodo por red, enviar y recibir mensajes tanto por TCP como por UDP y saber la id del nodo la cual es necesaria para que este anuncie que está vivo. Todo esto sabiendo solamente a quién y qué mensaje se quiere enviar, aislando al agente de toda la implementación.
Siguiendo con cliente.c, esta solo contiene la estructura ClienteConectado y su función crear. La estructura es utilizada para agregar a los FD que representan a los sockets información necesaria para la recepción de mensajes. Esta guarda el fd que representa al socket de la conexión, un entero que indica si la conexión fue por erlang (local del agente) o por un agente externo, un buffer de lectura (más su contador) donde se acumulan los mensajes que llegan partidos hasta rearmarlos, y —para la escritura no bloqueante— un buffer de salida (más su contador) y un mutex. Este buffer de salida es la contraparte de EPOLLOUT: si al enviar un mensaje el kernel no lo acepta entero (buffer de envío lleno), el remanente se guarda ahí y se activa EPOLLOUT; cuando el socket vuelve a tener lugar, el bucle drena el buffer. Así ningún hilo se bloquea escribiendo y no se pierden mensajes bajo backpressure. 
Por último, en controlador.c se realiza todo el procesamiento de los mensajes para luego integrar al agente con el gestor de recursos. En este se realiza un parseo del mensaje para distinguir qué petición tanto local como de red llegó y en base a esta poder determinar la decisión a llevar a cabo. Por ejemplo, si llegan requerimientos del erlang, se filtra qué mensaje este nos envió. Si es un job_request, nos quedamos con la información de a qué nodo se está pidiendo que recursos y en qué cantidad para luego poder armar el mensaje reserve a enviar al nodo obtenido. Si es un release, enviamos los release correspondientes a cada nodo que previamente nos había otorgado recursos. Siguiendo con el mismo funcionamiento y lógica para el resto de instrucciones tanto locales como de red

Un detalle importante del reactor es cómo se registran los descriptores en el epoll, porque de eso depende que el modelo multihilo sea correcto. Los sockets de escucha y los timers, que son descriptores compartidos y permanentes, se registran con EPOLLEXCLUSIVE, de modo que el kernel despierta a un solo hilo por evento y se evita el thundering herd. Los clientes conectados, en cambio, se registran con EPOLLONESHOT, lo que garantiza que a lo sumo un hilo procese a cada cliente a la vez: cuando termina de leer o de drenar su salida, el cliente se re-arma calculando la máscara EPOLLIN, más EPOLLOUT si todavía quedó salida pendiente, todo bajo su lock de salida para que un envío concurrente no pierda el aviso de EPOLLOUT. El envío por TCP recibe un fd crudo y encuentra el ClienteConectado correspondiente a través de un mapa global de fd a cliente, lo que le permite acceder a su buffer de salida; como cualquier hilo puede escribirle a un cliente, cada uno tiene su propio lock de salida.

En cuanto al ciclo de vida de las conexiones, el ClienteConectado es propiedad del bucle de epoll: lo crea al aceptar o conectar y lo destruye en el camino de desconexión. El registro de nodos guarda únicamente una referencia débil a la conexión (el campo conexion del nodo) que nunca cierra ni libera; cuando una conexión cacheada se cae, solo se pone ese campo en NULL, y el nodo sigue vivo hasta que lo elimine el timeout de 15 segundos. Esta separación de responsabilidades es la que evita los double free y los use-after-free entre el hilo que atiende el socket y el timer que limpia nodos.

Respecto de los protocolos que hablan por estos sockets, con Erlang, sobre la conexión local, el agente entiende GET_NODES —que responde con la tabla de nodos en el formato NODES IP:puerto:cpu:X:mem:Y:gpu:Z, separando varios nodos con punto y coma—, JOB_REQUEST seguido del id y de una lista de tokens @IP:recurso:cant para solicitar reservas en uno o varios nodos, y JOB_RELEASE con el id para liberar todo lo reservado cuando el job termina. Hacia Erlang el agente responde JOB_GRANTED cuando todos los nodos concedieron, JOB_DENIED cuando un nodo rechazó un pedido imposible, y JOB_TIMEOUT cuando la petición no se completó a tiempo. Entre agentes C, sobre el puerto público, el protocolo es RESERVE para pedir a otro nodo que reserve una cantidad de un recurso para un job, GRANTED y DENIED como respuestas a ese pedido, y RELEASE para cancelar o liberar una reserva, ya sea por rollback o por fin del job. Por último, el descubrimiento se hace por broadcast UDP con el mensaje ANNOUNCE, que incluye el puerto y los recursos disponibles y se emite cada 5 segundos; la IP la aporta el propio datagrama, y un nodo que no se anuncia durante 15 segundos se elimina del registro.
